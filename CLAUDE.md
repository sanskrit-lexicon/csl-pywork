# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

`csl-pywork` is one of three sibling repositories that together constitute the **Cologne Sanskrit Lexicon (CSL)** digital publishing pipeline. All three repos must be checked out as siblings in the same parent directory:

```
cologne/
  csl-orig/          # Source digitization text files (one per dictionary)
  csl-pywork/        # This repo — code generation for per-dictionary pywork scripts
  csl-websanlexicon/ # Web display code generation
```

The pipeline takes raw dictionary text files from `csl-orig` and generates XML, SQLite databases, web displays, and download archives for ~45 Sanskrit dictionaries.

**Active version is `v02/`. The `v00/` directory is obsolete.**

## Common Commands

All commands run from the `v02/` directory using bash (not PowerShell).

**Generate a single dictionary** (primary workflow):
```bash
cd v02
sh generate_dict.sh <dict> <outdir>
# e.g.: sh generate_dict.sh mw ../../MWScan/2020
# e.g.: sh generate_dict.sh acc tempparent/acc
```

**Generate all dictionaries** (Cologne server):
```bash
cd v02
sh redo_cologne_all.sh
```

**Generate all dictionaries** (XAMPP local):
```bash
cd v02
sh redo_xampp_all.sh
```

**Selective update** (only dictionaries changed in csl-orig since last run):
```bash
cd v02
sh redo_xampp_selective.sh
```

**Validate XML for a dictionary** (XAMPP layout):
```bash
cd v02
sh xmlchk_xampp.sh <dict>
```

**Regenerate abbreviation/bibliography/tooltip redo scripts** (updates `distinctfiles/<dict>/pywork/`):
```bash
cd v02
sh generate_ab_bib_ls.sh              # regenerate for all applicable dicts
sh generate_ab_bib_ls.sh <outdir/pywork>  # regenerate for one dict during generate_dict.sh
```

**Re-run only headword generation** (inside a generated pywork dir):
```bash
cd ../../XXXScan/2020/pywork
sh redo_hw.sh
```

**Re-run only XML + SQLite generation** (inside a generated pywork dir):
```bash
cd ../../XXXScan/2020/pywork
sh redo_xml.sh
```

## Architecture

### Code Generation Model

`csl-pywork` does not contain runnable dictionary scripts directly. Instead it generates them. For each dictionary, `generate_dict.sh` assembles a working directory (e.g. `../../MWScan/2020/`) containing three subdirectories:

- `orig/` — source digitization from `csl-orig`
- `pywork/` — scripts to build headwords, XML, and SQLite
- `web/` — display files (generated by `csl-websanlexicon/v02/generate_web.sh`)

### Key Files in `v02/`

| File | Role |
|---|---|
| `dictparms.py` | Defines all ~45 dictionaries: `dictlo` (lower), `dictup` (upper), `dictname`, `dictversion` |
| `inventory.txt` | Lists every file to place in `outdir/pywork/` and `outdir/downloads/`, with category C/T/CD/D |
| `inventory_orig.txt` | Lists source files to copy from `csl-orig` into `outdir/orig/` |
| `generate.py` | Core Python script: reads inventory, applies Mako templates, copies or deletes files |
| `generate_dict.sh` | Orchestrator: calls generate_orig → generate_pywork → generate_web → runs hw/xml/sqlite/downloads |
| `generate_orig.sh` | Runs `generate.py` with `inventory_orig.txt` to populate `orig/` |
| `generate_pywork.sh` | Runs `generate.py` with `inventory.txt` to populate `pywork/` |
| `generate_ab_bib_ls.sh` | Generates `redo.sh`/`redo_XXab.sh`/`xx.sql` scripts for abbreviation, tooltip, and bib tables |

### `inventory.txt` File Categories

- **C** — copy file verbatim from `makotemplates/`
- **T** — render file as a Mako template using dict-specific params from `dictparms.py`
- **CD** — copy file verbatim from `distinctfiles/<dict>/pywork/`
- **D** — delete file from target (used to clean up obsolete scripts)

Lines starting with `;` are comments. The first field is either `*` (all dicts) or a space-separated list of dict codes.

### `makotemplates/pywork/` — Template Sources

These are the source files that become per-dictionary pywork scripts after Mako rendering:

| Template | Generated output | Purpose |
|---|---|---|
| `hw.py` | `<dict>hw.py` | Extracts headwords from `orig/xxx.txt` → `xxxhw.txt` |
| `hw0.py`, `hw2.py` | `<dict>hw0.py`, `<dict>hw2.py` | Headword variant files |
| `hwparse.py` | `hwparse.py` | HW record parser used by make_xml.py |
| `make_xml.py` | `make_xml.py` | Builds `xxx.xml` from orig + headword files |
| `one.dtd` | `xxx.dtd` | DTD for XML validation (filename templated) |
| `redo_hw.sh` | `redo_hw.sh` | Runs the hw.py chain |
| `redo_xml.sh` | `redo_xml.sh` | Runs make_xml.py, xmllint, then redo_postxml.sh |
| `redo_postxml.sh` | `redo_postxml.sh` | Runs sqlite, webtc2, and optional ab/auth/keys steps |
| `sqlite/sqlite.py` | `sqlite/sqlite.py` | Creates `xxx.sqlite` from `xxx.xml` |
| `sqlite/sqlite_txt.py` | `sqlite/sqlite_txt.py` | Creates `xxx.sqlite` from tab-separated input |

### Mako Templating Special Cases

Templates use Mako syntax (`%if dictlo == 'mw':` / `%endif`, `${dictlo}`, etc.). However `make_xml.py` is a Python file that itself contains `<%s>` and `</%s>` XML tag literals which conflict with Mako syntax. `generate.py` handles this via `mako_preprocess()`/`mako_postprocess()` which swaps `<%s>` ↔ `!!!s!!!` and `</%s>` ↔ `!!!/s!!!` before/after template rendering. Also `##` is escaped as `!!!##!!!`.

### `distinctfiles/<dict>/pywork/` — Per-Dictionary Distinct Files

Files that are too divergent across dictionaries to be templated live here and are copied with the CD category. Notable:

- `distinctfiles/mw/pywork/` — MW has extensive extra files (mwkeys, westmwtab, whitmwtab, transcoder, mwauth)
- `distinctfiles/pw/pywork/` and `pwg/`, `pwkvn/` — have `make_xml_ls.py` (literary-source variant) and auth/bib files
- `distinctfiles/<dict>/pywork/<dict>ab/` — abbreviation input files for dicts with ab support
- `distinctfiles/<dict>/pywork/<dict>auth/` — literary source tooltip/bib files

### Per-Dictionary Generated Pipeline

After `generate_dict.sh` assembles `outdir/pywork/`, it runs these steps in sequence:

1. `sh redo_hw.sh` — `hw.py` reads `../orig/xxx.txt` + `hwextra/xxx_hwextra.txt` → `xxxhw.txt`; then `hw2.py` and `hw0.py` produce variant headword files
2. `sh redo_xml.sh` — `make_xml.py` reads orig + hw files → `xxx.xml`; xmllint validates against `xxx.dtd`; then calls `redo_postxml.sh`
3. `redo_postxml.sh` — copies `xxxheader.xml` to web/, runs `sqlite/redo.sh` (xml→sqlite), `webtc2/redo.sh`, and for applicable dicts: `<dict>ab/redo.sh`, `<dict>auth/redo.sh`, plus MW-specific westmwtab and whitmwtab
4. `downloads/redo_all.sh` — creates zip archives for txt, xml, and web

### Dictionary Naming Convention

- `dictlo` — lowercase code used in filenames: `mw`, `pw`, `skd`, `acc`
- `dictup` — uppercase code: `MW`, `PW`, `SKD`, `ACC`
- Directory layout on server: `XXXScan/2020/` (e.g. `MWScan/2020/`)
- MW and AP are special cases in many template conditionals due to extra entry types (H1–H4, H1A–H4E) and `e` tag

### Selective Update Mechanism (`redo_xampp_selective.sh`)

Tracks last-processed csl-orig commit in `csl-orig/v02/.xampp_last_run`. On each run, uses `git diff --name-only` to find `.txt` files changed since that commit, intersects with valid dict codes, and only regenerates affected dictionaries. Also propagates changes to cologne-stardict, csl-json, and csl-homepage repos.
